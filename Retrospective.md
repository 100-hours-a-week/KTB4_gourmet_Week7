# 회고 및 AI 사용 기록
# 회고
## 1. Fetch API로 프론트엔드와 백엔드를 연결하며 겪은 어려움

이번 과제에서는 기존에 화면만 구성되어 있던 프론트엔드와 Spring Boot 백엔드를 Fetch API로 연결하였다. 처음에는 버튼 클릭이나 입력값 검증처럼 화면 안에서만 동작하는 기능이 많았지만, 실제 API를 연결하면서 요청 URL, HTTP Method, Request Body, Response 구조를 모두 맞춰야 한다는 점이 어려웠다.

특히 회원가입, 로그인, 게시글 작성, 댓글 작성, 좋아요처럼 각 기능마다 요청 방식이 달라서 프론트엔드와 백엔드의 경로가 조금만 어긋나도 기능이 동작하지 않았다. 예를 들어 프론트에서는 `/posts/{postId}/comments`로 요청하는데 백엔드에서는 다른 경로를 기대하고 있으면 바로 오류가 발생했다. 이 과정을 통해 프론트엔드 화면 흐름과 백엔드 API 설계가 반드시 함께 맞아야 한다는 것을 느꼈다.

---

## 2. 이미지 업로드 방식을 구현하며 겪은 어려움

처음에는 이미지 파일명을 문자열로만 저장하면 된다고 생각했다. 하지만 실제로 이미지를 화면에 보여주려면 파일 자체가 서버에 저장되어야 하고, DB에는 이미지 경로를 저장해야 한다는 것을 알게 되었다. 그래서 기존 JSON 요청 방식에서 `multipart/form-data` 방식으로 변경하고, 백엔드에서는 `MultipartFile`을 사용해 파일을 저장하도록 구현하였다.

프로필 이미지는 `uploads/profile`, 게시글 이미지는 `uploads/posts`에 저장하도록 분리하였다. 또한 사용자가 올린 파일명을 그대로 저장하면 충돌이나 보안 문제가 생길 수 있어 UUID 기반 파일명으로 저장하도록 구성하였다. 이 과정에서 이미지 파일은 DB에 직접 저장하는 것이 아니라 서버에 저장하고, DB에는 접근 가능한 경로만 저장하는 구조를 이해하게 되었다.

---

## 3. 이미지 URL 공개 여부에 대해 고민한 점

이미지를 `/uploads/profile/...`, `/uploads/posts/...` 형태로 접근 가능하게 만들면서, 이 경로가 외부에서 접근 가능하면 위험하지 않은지 고민하였다. 처음에는 URL로 이미지가 보이는 것 자체가 보안상 문제가 되는 것처럼 느껴졌다.

하지만 프로필 이미지나 게시글 이미지는 사용자가 화면에서 볼 수 있어야 하는 공개 리소스이기 때문에, 브라우저가 접근 가능한 URL이 필요하다는 것을 이해했다. 대신 아무 파일이나 업로드되지 않도록 이미지 확장자와 Content-Type을 검사하고, 파일 크기를 제한하며, 저장 경로가 `uploads` 밖으로 벗어나지 않도록 검증하는 방식이 필요하다고 판단했다.

---

## 4. 게시글 이미지 여러 장 업로드와 순서 처리에 대해 고민한 점

처음에는 게시글 이미지 한 장만 고려했지만, 실제 게시글에는 여러 장의 이미지가 들어갈 수 있다는 점을 고려하게 되었다. 그래서 게시글 이미지 Entity를 따로 두고, 하나의 게시글에 여러 이미지가 연결될 수 있도록 구성하였다.

여러 이미지를 저장할 때는 업로드된 순서가 중요하다고 생각했다. 따라서 `sortOrder`를 사용해 이미지 순서를 저장하고, 첫 번째 이미지는 목록 화면에서 대표 썸네일로 사용할 수 있도록 했다. 이 과정을 통해 단순히 이미지를 저장하는 것뿐만 아니라, 목록 화면과 상세 화면에서 이미지를 어떻게 사용할지도 함께 고려해야 한다는 것을 느꼈다.

## 5. 게시글 목록에서 이미지 배치 방식을 고민한 점

게시글 목록에 이미지를 표시할 때 처음에는 이미지를 크게 보여주었지만, 그렇게 하니 게시글 목록이라기보다 이미지 피드처럼 보여서 어색했다. 게시판 목록에서는 제목, 좋아요 수, 댓글 수, 조회수, 작성자 정보가 먼저 보여야 한다고 생각했다.

그래서 목록에서는 첫 번째 이미지만 작은 썸네일로 보여주고, 상세 페이지에서 이미지를 크게 확인하는 방식이 더 적절하다고 판단했다. 최종적으로는 제목 오른쪽에 작은 썸네일을 두고, 아래에는 좋아요·댓글·조회수와 작성자·작성시간을 보여주는 형태로 정리하였다. 이를 통해 같은 이미지라도 목록 화면과 상세 화면에서 보여주는 방식이 달라야 한다는 것을 배웠다.

## 6. 회원정보 수정 화면에서 프로필 이미지 표시 문제를 겪은 점

회원정보 수정 화면에서는 현재 회원의 이메일, 닉네임, 프로필 이미지를 불러와 표시해야 했다. 이메일과 닉네임은 `GET /users/{userId}` 응답으로 쉽게 표시할 수 있었지만, 프로필 이미지는 HTML 구조와 JavaScript 처리 방식이 맞지 않아 어려움이 있었다.

특히 프로필 이미지 영역이 `<label>` 안에 `<img>`가 들어있는 구조였기 때문에, 이미지를 `backgroundImage`로 넣을지 `img src`로 넣을지 헷갈렸다. 결국 화면 구조에 맞게 원형 영역에 현재 프로필 이미지를 보여주고, 새 이미지를 선택하면 미리보기가 바뀌도록 정리하였다. 이 과정에서 DOM 구조와 CSS, JS 렌더링 방식이 서로 맞아야 화면이 제대로 동작한다는 것을 느꼈다.

---

## 7. JavaScript 파일 역할이 겹치며 생긴 문제

상단 프로필 메뉴는 여러 페이지에서 공통으로 사용되기 때문에 `profile-menu.js`로 분리하였다. 그런데 회원정보 수정 페이지의 `edit-profile.js`에서도 상단 프로필 버튼, 드롭다운, 로그아웃 관련 변수를 다시 선언하면서 문제가 발생했다.

같은 페이지에서 여러 JS 파일이 로드되면 전역 변수명이 충돌할 수 있고, 이 경우 스크립트 실행이 중단될 수 있다는 점을 알게 되었다. 이후 `profile-menu.js`는 상단 프로필 메뉴와 로그아웃만 담당하고, `edit-profile.js`는 회원정보 조회, 수정, 탈퇴만 담당하도록 역할을 분리하였다. 이를 통해 프론트엔드에서도 파일별 책임을 나누는 것이 중요하다는 것을 느꼈다.

## 8. 이벤트가 동작하지 않는 문제를 해결하며 느낀 점

이미지 선택, 수정하기 버튼, 드롭다운 클릭처럼 여러 이벤트를 연결하는 과정에서 예상대로 동작하지 않는 문제가 있었다. 처음에는 백엔드 API 문제라고 생각했지만, 실제로는 JavaScript가 중간에 오류로 멈추거나, 선택자가 실제 HTML 구조와 맞지 않아 이벤트가 연결되지 않는 경우가 있었다.

예를 들어 댓글 작성자 이미지 표시 문제에서는 HTML에는 `.writer-image` 클래스가 있었지만, JavaScript에서는 다른 클래스명을 찾고 있어 이미지가 표시되지 않았다. 이 문제를 해결하면서 이벤트나 DOM 조작이 안 될 때는 API 요청만 볼 것이 아니라, 콘솔 오류, JS 실행 여부, querySelector가 실제 요소를 찾고 있는지까지 함께 확인해야 한다는 것을 배웠다.

## 9. H2 DB와 localStorage 데이터가 어긋나는 문제

H2 Memory DB는 서버를 재시작하면 데이터가 초기화되지만, 브라우저의 localStorage는 그대로 남아 있었다. 이 때문에 이전에 저장된 `userId`가 남아 있는데, 서버 재시작 후 H2 DB의 데이터는 새로 생성되어 다른 회원을 가리키는 문제가 발생했다.

처음에는 회원정보 수정 화면에서 이메일과 닉네임이 이상하게 바뀐 것처럼 보여 코드 문제라고 생각했다. 하지만 확인해보니 localStorage에 남아 있는 userId와 H2 DB의 현재 데이터가 맞지 않는 것이 원인이었다. 이 과정을 통해 클라이언트에 저장된 로그인 정보와 서버 DB 상태가 다르면 예상하지 못한 화면 오류가 생길 수 있다는 것을 알게 되었다.

## 10. 좋아요와 조회수 기능을 연결하며 고민한 점

좋아요 기능은 단순히 좋아요 수만 증가시키는 것이 아니라, 사용자가 이미 좋아요를 눌렀는지 확인하고 다시 누르면 취소되는 토글 방식으로 구현해야 했다. 이를 위해 사용자와 게시글의 조합을 기준으로 좋아요 여부를 조회하고, 좋아요 수를 다시 계산해 응답하도록 구성하였다.

조회수 역시 상세 페이지에 들어갈 때마다 무조건 증가하면 새로고침만 해도 조회수가 계속 오르는 문제가 생긴다. 그래서 사용자별로 한 번만 조회수가 증가하도록 별도의 조회 기록을 저장하는 방식으로 구현하였다. 이를 통해 단순한 숫자 증가 기능도 실제 서비스처럼 만들려면 중복 처리 기준이 필요하다는 것을 느꼈다.

## 11. 게시글 수정에서 이미지 변경 방식을 고민한 점

게시글 수정에서는 제목과 내용만 수정할 수도 있고, 이미지를 새로 선택해 교체할 수도 있어야 했다. 그래서 새 이미지를 선택하지 않으면 기존 이미지를 유지하고, 새 이미지를 선택하면 기존 이미지를 삭제한 뒤 새 이미지로 교체하는 방식으로 구현하였다.

이 과정에서 수정 API도 기존 JSON 방식으로는 이미지 파일을 보낼 수 없기 때문에 `multipart/form-data` 방식으로 변경해야 했다. 회원정보 수정과 게시글 수정 모두 이미지가 포함될 수 있는 요청은 FormData로 처리해야 한다는 점을 다시 확인했다.

# AI 사용 기록 문서

1. 이미지 업로드 방식에 대한 질문과 개선

처음에는 프로필 이미지나 게시글 이미지를 DB에 직접 저장해야 하는지, 파일명만 저장해도 되는지 헷갈렸다. 그래서 AI에게 “H2를 사용하는데 이미지 선택을 해도 실제 DB나 화면에 안 뜨는 경우에는 어떻게 저장하는 것이 맞는지” 질문하였다.

AI 답변을 통해 이미지 파일 자체는 서버의 `uploads` 폴더에 저장하고, DB에는 `/uploads/profile/...`, `/uploads/posts/...` 같은 이미지 경로만 저장하는 방식이 적절하다는 것을 확인하였다. 이를 바탕으로 프로필 이미지는 `uploads/profile`, 게시글 이미지는 `uploads/posts`에 저장하도록 구조를 나누었다.

---

2. 이미지 URL 공개 여부에 대한 질문과 개선

이미지 저장 후 `http://localhost:8080/uploads/profile/test.png`처럼 URL로 접근되는 것이 위험하지 않은지 궁금했다. 그래서 AI에게 “이런 식으로 이미지가 뜨면 외부에서 접근 가능하거나 조작되어 위험하지 않은지” 질문하였다.

답변을 통해 프로필 이미지나 게시글 이미지는 화면에 표시되어야 하는 공개 리소스이므로 URL 접근 자체는 필요하다는 것을 알게 되었다. 대신 아무 파일이나 업로드되지 않도록 확장자, Content-Type, 파일 크기, 저장 경로를 검증해야 한다는 점을 확인하였다. 이후 `FileStorageService`에서 이미지 파일만 허용하고, UUID 파일명으로 저장하며, 경로 조작을 방지하는 방향으로 개선하였다.

---

3. 회원가입과 회원정보 수정에서 이미지 전송 방식에 대한 질문과 개선

기존 회원가입과 회원정보 수정은 JSON 방식으로 요청을 보내고 있었다. 하지만 이미지 파일을 함께 보내야 하면서 기존 방식으로는 실제 파일이 전달되지 않는 문제가 있었다. 그래서 AI에게 “프로필 이미지를 실제 저장하려면 회원가입과 회원정보수정 API를 어떻게 바꿔야 하는지” 질문하였다.

AI 답변을 참고해 JSON 요청 대신 `FormData`를 사용하도록 변경하였다. 백엔드에서는 `@RequestParam`으로 닉네임 등의 문자열 값을 받고, `@RequestPart MultipartFile`로 이미지 파일을 받도록 수정하였다. 이를 통해 프로필 이미지가 실제 서버 폴더에 저장되고, DB에는 이미지 경로가 저장되도록 개선하였다.

---

4. 게시글 이미지 여러 장 업로드에 대한 질문과 개선

처음에는 게시글 이미지 한 장만 고려했지만, 여러 장 이미지가 노출될 수 있다는 요구를 생각하면서 구조가 부족하다고 느꼈다. 그래서 AI에게 “게시글 이미지 여러 장을 저장하려면 어떻게 구성하는 것이 좋은지” 질문하였다.

답변을 바탕으로 게시글 이미지를 `PostImage` 엔티티로 분리하고, 하나의 게시글에 여러 이미지가 연결될 수 있도록 구성하였다. 또한 여러 이미지가 업로드될 때 순서가 필요하다고 판단하여 `sortOrder`를 저장하고, 첫 번째 이미지는 대표 썸네일로 사용할 수 있도록 개선하였다.

---

5. 게시글 목록 이미지 배치에 대한 질문과 개선

게시글 목록에 이미지를 크게 보여주었더니 목록 화면이 게시판이라기보다 이미지 피드처럼 보여 어색했다. 그래서 AI에게 “게시글 목록에서 이미지가 들어가면 사진만 너무 크게 보이는데 어떻게 배치하는 것이 적절한지” 질문하였다.

처음에는 큰 이미지 형태로 표시했지만, 실제 화면을 보며 다시 고민했고 “제목 오른쪽에 작은 썸네일을 두고, 아래에는 좋아요·댓글·조회수와 작성자 정보를 배치하는 방식”이 더 적절하다고 판단하였다. AI와 배치 방향을 조정하면서 목록에서는 첫 번째 이미지만 작은 썸네일로 보여주고, 상세 페이지에서 이미지를 크게 보여주는 방식으로 개선하였다.

---

6. 게시글 상세 이미지 표시 방식에 대한 질문과 개선

게시글 작성 시 이미지는 저장되었지만, 상세 조회 화면에서 이미지가 보이지 않는 문제가 있었다. 그래서 AI에게 “저장된 게시글 이미지를 상세 페이지에서 표시하려면 어떤 응답 구조와 렌더링이 필요한지” 질문하였다.

답변을 참고해 `PostResponseDto`에 `imageUrls`를 추가하고, `PostService`에서 게시글 이미지 목록을 조회해 응답에 포함하도록 수정하였다. 프론트엔드에서는 `post.imageUrls`를 순회하면서 이미지 태그를 생성해 상세 페이지에 표시하도록 개선하였다.

---

7. 게시글 수정에서 이미지 변경 방식에 대한 질문과 개선

게시글 수정 시 제목과 내용만 수정할 수도 있고, 이미지를 새로 선택해 변경할 수도 있어야 했다. 그래서 AI에게 “게시글 수정에서 이미지를 선택하지 않으면 기존 이미지를 유지하고, 새 이미지를 선택하면 교체하려면 어떻게 해야 하는지” 질문하였다.

답변을 바탕으로 게시글 수정 API도 `multipart/form-data` 방식으로 변경하였다. 새 이미지가 없는 경우에는 기존 이미지를 유지하고, 새 이미지가 있는 경우에는 기존 이미지 정보를 삭제한 뒤 새 이미지를 저장하는 방식으로 개선하였다.

8. 댓글 작성자 프로필 이미지가 표시되지 않는 문제 해결

댓글 작성자 프로필 이미지가 화면에 표시되지 않아 원인을 찾기 어려웠다. 그래서 AI에게 댓글 렌더링 함수 일부를 보여주고 “댓글에 회원 사진이 안 나온다”고 질문하였다.

AI와 함께 코드를 확인하면서 백엔드 응답 문제가 아니라 프론트엔드 DOM 선택자 문제라는 것을 알게 되었다. 실제 HTML에서는 `.writer-image` 클래스를 사용하고 있었지만, JavaScript에서는 다른 클래스명을 찾고 있었다. 선택자를 실제 DOM 구조에 맞게 수정하여 댓글 작성자 프로필 이미지가 정상적으로 표시되도록 개선하였다.

9. 회원정보 수정 화면에서 이메일, 닉네임, 프로필 이미지가 안 뜨는 문제 해결

회원정보 수정 화면에서 이메일과 닉네임이 표시되지 않고, 프로필 이미지도 회색으로 남는 문제가 있었다. 그래서 AI에게 `edit-profile.html`, `edit-profile.js`, `UserController`, `UserService`, `UserResponseDto`, `User` 코드를 함께 확인해달라고 요청하였다.

AI와 함께 확인한 결과 백엔드에서는 `GET /users/{userId}` 응답으로 `email`, `nickname`, `profileImage`를 정상적으로 내려주고 있었고, 문제는 프론트엔드 JS 파일 역할이 겹치는 데 있었다. `profile-menu.js`와 `edit-profile.js`에서 상단 프로필 관련 변수를 중복 선언하면서 스크립트 실행이 꼬일 수 있었다. 이후 `profile-menu.js`는 상단 메뉴와 로그아웃만 담당하고, `edit-profile.js`는 회원정보 조회·수정·탈퇴만 담당하도록 역할을 분리해 개선하였다.

---

10. H2 DB와 localStorage 데이터 불일치 문제 확인

테스트 중 회원정보 수정 화면에서 예상과 다른 이메일과 닉네임이 표시되는 일이 있었다. 그래서 AI에게 “왜 이메일과 닉네임이 다른 값으로 자동으로 바뀌는지” 질문하였다.

답변을 통해 H2 Memory DB는 서버 재시작 시 데이터가 초기화되지만, 브라우저 localStorage는 그대로 남아 있다는 점을 확인하였다. 이 때문에 이전에 저장된 `userId`가 현재 DB의 다른 회원을 가리킬 수 있었다. 이후 테스트 중 데이터가 꼬이면 `localStorage.clear()` 후 다시 로그인하여 확인하는 방식으로 문제를 해결하였다.

11. 좋아요와 조회수 기능 구현 방향에 대한 질문과 개선

좋아요 기능은 단순히 숫자를 증가시키는 것이 아니라, 사용자가 이미 좋아요를 눌렀는지 확인하고 다시 누르면 취소되는 토글 방식이 필요했다. 그래서 AI에게 “좋아요를 토글 방식으로 구현하려면 API와 Repository를 어떻게 구성하면 좋은지” 질문하였다.

답변을 참고해 사용자와 게시글 조합을 기준으로 좋아요 여부를 확인하고, 좋아요가 있으면 삭제하고 없으면 생성하는 방식으로 구현하였다. 조회수 역시 상세 페이지에 들어갈 때마다 무조건 증가하지 않도록, 사용자별 조회 기록을 저장하고 처음 조회한 경우에만 조회수가 증가하도록 개선하였다.

12. GitHub Push 오류 해결 과정

과제 완료 후 GitHub에 push하는 과정에서 `fetch first` 오류가 발생했다. 처음에는 왜 push가 거절되는지 알기 어려워 AI에게 오류 화면을 보여주고 원인을 질문하였다.

답변을 통해 GitHub 원격 저장소에 로컬에는 없는 README 등의 커밋이 이미 있어서 발생한 문제라는 것을 알게 되었다. 이후 `git pull origin main --rebase`로 원격 내용을 먼저 반영한 뒤 다시 push하여 해결하였다.

13. 전체적인 AI 활용 방식

이번 과제에서 AI는 코드를 대신 작성하게 하는 용도보다는, 구현 과정에서 헷갈리는 개념을 확인하고 문제 원인을 좁히는 데 활용하였다. 특히 이미지 업로드 구조, FormData 요청 방식, 게시글 이미지 배치, JS 파일 역할 분리, H2와 localStorage 상태 차이처럼 직접 판단이 필요한 부분에서 질문을 통해 방향을 잡았다.

---

AI 답변을 그대로 적용하기보다는 실제 화면과 코드 상태를 확인하면서 맞지 않는 부분은 다시 질문하고 수정하였다. 이를 통해 단순히 기능을 완성하는 것뿐만 아니라, 프론트엔드와 백엔드가 어떤 흐름으로 연결되는지 더 구체적으로 이해할 수 있었다.
